# Goal Options View — Implementation Summary

## What Was Built

A new "Options" view for the goal detail modal, accessible via a toggle button in the persistent header next to the SMART indicator. The view provides four goal management operations.

---

## Features Implemented

### 1. Copy Goal
Creates a duplicate of the current goal with all configuration cloned — name (prefixed "Copy of"), description, deadline, relevance statement, parent association, behavioral flags, targets, and activity/group associations. Children and completion state are not copied. After copying, the user is navigated directly to the new goal in edit mode.

### 2. Freeze / Unfreeze Goal
Toggles a new `frozen` boolean field on the goal. Frozen goals:
- Cannot be manually completed (enforced both in the UI and backend service layer)
- Cannot be auto-completed via target evaluation
- Display a "Frozen" badge in the header (blue/gray)
- Show a grayed-out "Mark Complete" button with reduced opacity
- Can still be viewed, edited (metadata), and deleted

### 3. Move Goal
Re-parents the goal under a different goal in the tree. Includes:
- Circular reference prevention (walk up the ancestor chain)
- Level rank compatibility validation (goal must sit below its new parent in rank)
- Parent capacity validation (respects `max_children` on GoalLevel)
- Searchable, depth-indented goal picker with the current parent highlighted
- Option to remove parent entirely (make root-level within its tier)

### 4. Convert Goal Level
Changes the goal's level/type. Includes:
- Rank validation against parent (new level must be below parent's rank)
- Rank validation against children (new level must be above all children's ranks)
- Visual level selector pulled from `GoalLevelsContext`, with current level highlighted

## Self-Assessment

### Grade: B+

**What went well:**

- **Backend is solid.** The service methods follow existing patterns precisely — same signature, same error tuple return, same ownership validation. The freeze enforcement is placed correctly in both the manual completion path and the target auto-complete path. Circular reference prevention in `move_goal` and bi-directional rank validation in `convert_goal_level` are both correct.
- **Migration is clean.** Scoped to just the two new columns, unrelated autogenerated noise was removed.
- **Header toggle behavior** is exactly as specced — persists across both views, toggles text between "Options" and "Back to Goal Details", only appears in view mode (not create mode).
- **ViewState pattern** is followed correctly — `goal-options` slots cleanly into the existing `if/else if` routing chain in `GoalDetailModal`.
- **API client** follows the exact naming and parameter conventions of the existing entries.

**Where it falls short (why not A):**

- **`GoalOptionsView` uses `fractalApi` directly** rather than going through `GoalsContext` or a dedicated hook. The existing pattern for heavier mutations (completion, associations) uses React Query's `useMutation` with `onSuccess`/`onError`. The options view uses a simpler try/catch + `useState(loading)` pattern instead, which is inconsistent with the rest of the modal. This works but would be harder to extend with optimistic updates later.

- **The move sub-view uses `treeData` passed as a prop**, which is the full fractal tree already loaded by the parent page. This is efficient (no extra fetch), but `treeData` shape assumptions (`node.id`, `node.name`, `node.children`) are inferred rather than validated against the actual tree structure coming from the backend. If the tree format differs (e.g., nodes have `attributes.id` instead of `id`), the flat list would silently render empty.

- **Frozen goal in `GoalViewMode`** only disables `canShowManual` — it does not prevent the completion modal from opening when the goal is already completed (the "uncomplete" path). A frozen completed goal can currently be uncompleted from the UI since the frozen guard only blocks forward completion, not reversal. Whether this is intentional is ambiguous, but it's worth a decision.

- **`GoalConvertSubView` doesn't pre-filter incompatible levels client-side** — it relies on the backend to reject invalid conversions with an error toast. This is functionally correct but creates a slightly jarring UX where an apparently selectable level produces an error after the click rather than being visually grayed out upfront. Computing incompatibility client-side would require knowing the parent and children level ranks, which are available in `treeData` but would require a bit more wiring.

- **Copy Goal's `onGoalSelect` callback** assumes the calling context knows how to open a new goal in edit mode. This works when the modal is used from `FractalGoals.jsx`, but if `GoalDetailModal` is used in other contexts where `onGoalSelect` navigates differently, the "open in edit mode" side effect (`setIsEditing(true)`) happens in the options component without the parent knowing, which could cause state inconsistency.

**Overall:** The implementation is complete, builds cleanly, follows the codebase's patterns well in the backend, and delivers all four requested features. The frontend could be more consistent with the React Query mutation pattern used elsewhere in the modal, and a few edge cases around frozen state and level selection UX could be tightened.

## S-Rank Hardening Review

The current implementation is functionally close, but it is not yet aligned with the intended modal UX or the final hierarchy rules.

### Required Corrections

1. **Move Goal must use hierarchy context, not a flat list.**
   The selector should let the user switch to a different parent on the same parent tier as the current parent, with ancestor context visible.

2. **The goal detail header must persist across detail and options views.**
   Options should feel like a sub-view inside the same modal, not a separate screen with a different top section.

3. **Options actions need to match the goal detail modal styling.**
   The current outlined list works, but the visual language is weaker than the existing action grid and section cards in the detail view.

4. **Convert Level must exclude the root tier.**
   A goal cannot be converted to the same level as the `root_id` goal. In practice this means the Ultimate Goal tier must never appear as a valid conversion target for descendants.

5. **Header status must show `Active`, `Inactive`, or `Frozen`.**
   Frozen remains the strongest state. Active/inactive should align with the graph’s recent-activity branch logic when that data is available.

6. **Frozen goals must dim under `Fade inactive branches`.**
   This should apply even when there is no recent-activity evidence for the rest of the tree.

7. **Testing coverage must move from smoke-level to regression-level.**
   The current implementation lacks enough contract coverage for move/freeze/convert behavior and not enough UI coverage for the shared-header/options flow.

## S-Rank Implementation Plan

### Backend

- Tighten `move_goal` so reparenting only allows a new parent on the same tier as the current parent.
- Reject root-tier conversions in `convert_goal_level`.
- Keep frozen-goal completion blocking enforced at the service layer.

### Frontend Modal UX

- Render one shared header shell for both goal detail and options.
- Add a header status pill for `Active`, `Inactive`, and `Frozen`.
- Restyle options actions to match the modal’s existing card/button language.
- Disable frozen completion interactions in the detail view.

### Hierarchy + Conversion UX

- Replace the flat move picker with a hierarchy-based selector that preserves ancestor context.
- Filter selectable move targets to valid same-tier parents only.
- Filter convert-level options client-side so the root tier and other incompatible levels never appear as valid actions.

### Graph Behavior

- Pass frozen state through normalized goal nodes into the flow graph presentation layer.
- Dim frozen nodes and connecting edges when `Fade inactive branches` is enabled.

### Test Coverage

- Add service tests for frozen completion blocking, same-tier move validation, and root-tier conversion rejection.
- Add API coverage for freeze, move, and convert routes.
- Add modal regression coverage for the persistent header across options/detail.
- Add graph presentation coverage for frozen-node dimming.
